# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`vn-address-kit` is a TypeScript-first library + CLI for Vietnam's 2025 two-level administrative reform: it converts/searches/validates addresses from the legacy **3-level** model (`province → district → ward`) to the new **2-level** model (`province → ward`, district removed). Every conversion is deterministic and auditable — results carry a `confidence` score, a `strategy`, `warnings`, and `candidates` rather than forcing a single answer.

## Commands

```bash
npm run build          # tsup → dist/ (dual ESM+CJS, .d.ts, sourcemaps)
npm run dev            # tsup --watch
npm test               # vitest run (one-shot)
npm run test:watch     # vitest watch
npm run typecheck      # tsc --noEmit  (lint is an alias for this — there is no ESLint)
npm run release:check  # typecheck + test + build + npm audit + npm pack --dry-run
```

Run a single test file or test by name:

```bash
npx vitest run tests/convert-text.test.ts
npx vitest run -t "ambiguous"
```

Run the CLI locally without building (tsx-style entry is bundled; use the built binary after `npm run build`):

```bash
node dist/cli/index.js convert "123 Le Loi, P Loc Tho, TP Nha Trang, Khanh Hoa" --json
```

## Architecture

**Public API is a deliberate two-layer barrel.** `src/index.ts` is the published entry point; it re-exports everything from `src/public.ts` (which barrels the feature modules) plus the data loader and the `src/types/*` types. When adding a public function, wire it through `public.ts` → `index.ts`, not directly. `tsup.config.ts` builds two entry points: `index` (the library) and `cli/index` (the binary).

**Feature modules** under `src/` each own one verb and expose an `index.ts` barrel:

- `normalize/` — foundational text layer (`normalizeText`, `removeVietnameseTones`, `normalizeAdministrativePrefix`). Everything else matches through these, so behavior changes here ripple across search/convert/validate.
- `search/` — fuzzy lookup of provinces/wards (Levenshtein + scoring).
- `validate/` — hierarchy/address validation against the dataset.
- `convert/` — **the core engine** (see below).
- `data/` — the dataset access layer.

**Conversion flow.** Free text and structured input converge on one engine:

```
convertAddressText(text) → parseAddress(text) → convertOldToNew(structured)
convertOldToNew(structured) ← the single source of truth
convertBatch([...]) maps each item through one of the two above
```

`convertOldToNew` (`src/convert/convert-old-to-new.ts`) walks legacy province → district → ward, then looks up `mappings.json` (old codes → new codes) to build a candidate list. **Outcome is decided by candidate count:** exactly one unique candidate → `success`; multiple → ambiguous failure with candidates returned for review; none → fall back to province-level candidate suggestions or `failed`. It never guesses a single answer when ambiguous.

**Confidence is centralized** in `src/convert/confidence.ts`: each `MatchStrategy` has a base score (`BASE_SCORES`), and any result with more than one candidate is capped at `0.7`. Add a new matching path by adding a `MatchStrategy` (in `src/types/result.ts`) and its base score here — don't scatter magic numbers in the engine.

**Data layer.** `src/data/loader.ts` imports the bundled JSON in `src/data/official/` (`current-provinces`, `current-wards`, `legacy-provinces/districts/wards`, `mappings`) and exposes typed getters. This is the **official 2025 two-level dataset** (34 provinces / 3,321 wards), generated by `scripts/build-dataset.py` (`npm run build:data`) from two authoritative GSO files in `data/source/`: `conversion-table.xlsx` (the 2025 baseline) and `doi-chieu-2024-09-01.xls` (source of `LegacyWard.formerNames` — pre-2025 ward names like "Phước Tiến", see `docs/adr/0001` and `docs/adr/0003`). Only the generated JSON is read at runtime, so the core stays dependency-free; the parser is a dev-only Python script (openpyxl + xlrd). To update for a future reform, replace the source files and re-run `build:data`; the loader signature stays the same.

**CLI** (`src/cli/`) is Commander-based. `index.ts` registers commands from `commands/` (`convert`, `migrate`, `search`, `version`); `utils/csv.ts` and `utils/output.ts` handle CSV migration I/O and JSON/pretty formatting.

**Tests** (`tests/*.test.ts`) are Vitest and mirror the feature modules one-to-one.

## Conventions

- Pure ESM package (`"type": "module"`); imports use explicit relative paths without extensions, resolved by tsup.
- Public-facing functions return result objects (`success`/`confidence`/`warnings`/`candidates`) rather than throwing for "no match" cases — preserve this when extending the engine.
- Only runtime dependency is `commander`; keep the library core dependency-free.
